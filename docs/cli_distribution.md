# CLI Distribution

reVault publishes prebuilt `lockbox` and `lbx` executables so installing the
CLI does not require Rust or Cargo. CLI releases are independent of native API
releases and use tags of the form `revault_cli-vMAJOR.MINOR.PATCH`.

## Release Outputs

The generated CLI release workflow builds and attaches these artifacts to a
GitHub release:

| Platform | Architectures | Artifacts |
| --- | --- | --- |
| Linux (glibc) | x86-64, ARM64 | `.tar.gz`, SHA-256 checksum |
| macOS | Intel, Apple silicon | `.tar.gz`, SHA-256 checksum |
| Windows | x86-64 | `.zip`, `.msi`, SHA-256 checksums |
| Unix installer | supported Linux and macOS targets | `revault_cli-installer.sh` |
| Windows installer | x86-64 | `revault_cli-installer.ps1` |

GitHub build provenance attestations are generated for release artifacts. The
script installers place `lockbox` and `lbx` in `~/.revault/bin` and arrange for
that directory to be available on `PATH`.

### Linux compatibility

Linux binaries target glibc and dynamically link to `libdbus-1`. Mainstream
desktop and server distributions normally provide it. Minimal installations
may need the runtime package first:

```bash
# Debian and Ubuntu
sudo apt-get install libdbus-1-3

# Fedora and RHEL family
sudo dnf install dbus-libs
```

Alpine/musl is not currently a supported binary target. Adding it requires a
separate design for the CLI's D-Bus sleep/session integration; relabelling the
current glibc binary as portable would not make it work on musl.

## Installing a Published Version

Use the versioned release URL because this repository also publishes native
API releases. GitHub's repository-wide `latest` release can therefore refer to
a different release stream.

On Linux or macOS:

```bash
curl --proto '=https' --tlsv1.2 -LsSf \
  https://github.com/onepub-dev/reVault/releases/download/revault_cli-v0.0.3/revault_cli-installer.sh \
  | sh
```

On Windows PowerShell:

```powershell
irm https://github.com/onepub-dev/reVault/releases/download/revault_cli-v0.0.3/revault_cli-installer.ps1 | iex
```

Users who do not want to pipe a remote script into a shell can download the
matching archive or MSI and its checksum from the same GitHub release. Verify
the installation with:

```text
lockbox --version
```

The same release also publishes `revault_cli` and its workspace dependency
chain to crates.io. `cargo install revault_cli` therefore remains available as
a developer-oriented fallback, but is not the primary end-user installation
path because it requires a Rust toolchain.

## Publishing a CLI Release

1. Update `rust/revault_cli/Cargo.toml` and
   `rust/revault_cli/CHANGELOG.md` to the release version.
2. If public APIs changed, update the language bindings and migrations required
   by `rust/AGENTS.md` before releasing.
3. Run the required repository checks:

   ```bash
   cd rust
   cargo xtask check-required
   dist plan --tag revault_cli-v0.0.3
   ```

4. Commit the version and changelog, then create and push the matching tag:

   ```bash
   git tag revault_cli-v0.0.3
   git push origin revault_cli-v0.0.3
   ```

5. Confirm that all five target builds, the global installer job, artifact
   attestations, the crates.io publish job, and the GitHub release complete
   successfully. The crates are published in dependency order; a retry skips
   any package version already visible on crates.io.
6. Install from each published installer and run at least `lockbox --version`,
   `lockbox --help`, and `lockbox doctor` on Linux, macOS, and Windows.

The release workflow and MSI definition are generated by the pinned version of
`dist`. After editing distribution metadata, regenerate them from `rust/`:

```bash
dist generate
```

The crates.io publication is a cargo-dist custom publish job. Before the first
release, configure crates.io trusted publishing for every crate in the CLI
dependency set, using this repository, the `revault-cli-crates.yml` workflow,
and the `release` GitHub environment.

## Package Manager Rollout

The GitHub artifacts are the foundation for package-manager channels. Add
channels in this order:

1. **Organization Homebrew tap** for macOS and Linux. Use a custom tap such as
   `onepub-dev/homebrew-tap` and publish a binary formula from the release
   archives. The source-available reVault license is not eligible for
   `homebrew/core`, whose policy requires a DFSG-compatible licence and source
   builds for native software.
2. **WinGet** using the x86-64 MSI. Sign the executable and MSI first, then
   submit versioned manifests to `microsoft/winget-pkgs`. WinGet manifests must
   refer to an immutable public release URL and checksum.
3. **Scoop bucket** using the Windows ZIP. This is low maintenance once the
   release URL and hash are stable, and provides a useful portable alternative
   to the system-wide MSI.
4. **Debian and RPM packages** for users who require native package ownership
   and upgrades. Declare `libdbus-1-3` on Debian-family packages and `dbus-libs`
   on RPM-family packages. Publish them through an APT/RPM repository only
   after installation, upgrade, and removal tests exist.
5. **Chocolatey, AUR `-bin`, and Nix binary packaging** after demand justifies
   maintaining additional manifests. These should download the prebuilt
   artifacts rather than compile Rust on the user's machine.

The Homebrew tap, WinGet repository, Scoop bucket, and Linux package repository
are external publication surfaces. Their initial repositories, credentials,
and review submissions are intentionally not created by the binary release
workflow.

## Signing Before Broad Promotion

GitHub attestations establish build provenance, but they do not replace
platform code signing.

- Sign the Windows executables and MSI with an Authenticode certificate before
  promoting the MSI through WinGet. Unsigned builds are suitable for release
  testing but can trigger SmartScreen warnings.
- Sign macOS executables with a Developer ID Application certificate and
  notarize the distributed archive before presenting browser downloads as the
  preferred macOS installation path.
- Keep checksums and provenance attestations enabled for every platform,
  including signed artifacts.
